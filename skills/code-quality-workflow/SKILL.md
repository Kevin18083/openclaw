# 代码质量保障工作流

> **原则**：写代码前多思考 30 秒，胜过调试 30 分钟。

---

## 📋 五步流程

```
规划 → 编码 → 自检 → 复检 → 审核
```

每一步都有检查清单，**不允许跳过**！

---

## 第一步：规划 🧠

**触发时机**：收到编码任务后，**写第一行代码前**

### 0.1 前置检查（必须先做！）

**打开 `bug-lessons.md`，重读一遍所有教训！**

```
□ 已阅读 bug-lessons.md 所有条目
□ 本次任务相关的教训已记在脑子里
□ 提醒自己：同一个错误，绝不犯第二次！
```

### 1.1 理解需求

- [ ] 用户真正想要什么？（复述需求，确认理解）
- [ ] 输入/输出是什么？
- [ ] 边界条件有哪些？（空值、错误、极端情况）
- [ ] 有什么**不该做**的？（限制条件）

### 1.2 探索上下文

- [ ] 相关代码文件在哪里？（先读，再写）
- [ ] 现有的代码风格/约定是什么？
- [ ] 有没有类似的参考实现？
- [ ] 依赖是否已安装？

### 1.3 设计方案

向用户呈现（复杂任务需要用户确认）：

```
目标：[一句话]
方法：[用什么技术/模式]
改动范围：[哪些文件]
风险点：[可能出问题的地方]
替代方案：[有其他做法吗]
```

### 1.4 输出

简单的说："好的，我明白了，开始写代码"

复杂的说："我的计划是 XXX，对吗？"（等用户确认）

---

## 第二步：编码 💻

**触发时机**：规划完成/用户确认后

### 2.1 编码原则

| 原则 | 说明 |
|------|------|
| **单一职责** | 一个函数只做一件事 |
| **命名清晰** | 名字说明意图，不是缩写 |
| **错误处理** | 不吞异常，不静默失败 |
| **类型安全** | 能用 TypeScript 就不用 JS |
| **配置外置** | 魔法数字/字符串提取为常量 |

### 2.2 安全编码

**永远不要**：
- ❌ 硬编码密码/API 密钥
- ❌ 直接拼接 SQL 字符串
- ❌ 相信用户输入（始终验证）
- ❌ 用 `eval()` 解析用户数据
- ❌ 写死文件路径（用路径拼接）

### 2.3 日志与调试

```javascript
// ✅ 好的日志
logger.info('用户登录成功', { userId, timestamp })
logger.error('数据库连接失败', { error: e.message, retryCount })

// ❌ 坏的日志
console.log('111')
console.log(data)
```

### 2.4 提交前检查

- [ ] 代码能编译/运行吗？
- [ ] 有 lint 错误吗？
- [ ] 有 console.log 调试代码要删吗？
- [ ] 有注释掉的代码要清理吗？

---

## 第三步：自检 🔍

**触发时机**：代码写完，**告诉用户前**

### 3.1 功能检查

- [ ] 代码真的实现了需求吗？（逐条对照）
- [ ] 有没有 scope creep（多写了不必要功能）？
- [ ] 有没有漏掉什么？

### 3.2 边界条件检查

| 场景 | 检查项 |
|------|--------|
| 空值 | 输入为空/undefined/null 时怎么办？ |
| 错误 | API 失败/网络超时时怎么办？ |
| 极端 | 数据量极大/极小时怎么办？ |
| 并发 | 同时多次调用会怎样？ |

### 3.3 代码质量检查

- [ ] 有重复代码吗？（DRY）
- [ ] 函数超过 50 行吗？（拆分）
- [ ] 嵌套超过 3 层吗？（重构）
- [ ] 参数超过 5 个吗？（用对象）

### 3.4 安全检查

- [ ] 有敏感信息要移除吗？（密钥、密码）
- [ ] 有用户输入要验证吗？
- [ ] 有路径遍历风险吗？（../攻击）
- [ ] 有注入风险吗？（SQL、XSS）

---

## 第四步：复检 📖

**触发时机**：自检完成后

### 4.1 重新读代码

假装是别人写的代码，问自己：
- [ ] 这代码看得懂吗？
- [ ] 命名清晰吗？
- [ ] 注释有必要吗？
- [ ] 逻辑有漏洞吗？

### 4.2 追踪执行路径

选一个典型请求，**手动模拟**执行：
```
输入 → 函数 A → 函数 B → 数据库 → 返回
```

每一步问：
- [ ] 数据格式对吗？
- [ ] 可能抛异常吗？
- [ ] 异常有人处理吗？

### 4.3 测试覆盖检查

有测试吗？
- [ ] 正常情况测试
- [ ] 边界条件测试
- [ ] 错误情况测试

没测试？至少手动验证一下！

---

## 第五步：审核 🎯

**触发时机**：复检完成，**准备回复用户前**

### 5.1 最终检查清单

- [ ] 代码能运行（跑过吗？）
- [ ] 测试能通过（如果有的话）
- [ ] 没有明显 bug
- [ ] 没有安全风险
- [ ] 没有性能陷阱

### 5.2 回复用户

回复时包含：
```
✅ 做了什么
📁 改了哪些文件
⚠️ 注意事项（如果有的话）
🧪 如何测试/验证
```

### 5.3 后续建议

如果有改进空间，告诉用户：
```
💡 未来可以改进：
- XXX 可以优化性能
- YYY 可以加缓存
- ZZZ 可以写测试
```

## 📊 快速检查清单（打印版）

**贴显示器旁边，每次写完代码对照**！

```
□ 规划：理解需求、探索上下文、设计方案
□ 编码：单一职责、命名清晰、错误处理、安全编码
□ 自检：功能、边界、质量、安全
□ 复检：重读代码、追踪路径、测试覆盖
□ 审核：最终检查、回复用户、后续建议
```

## 🚨 红旗警告（看到就停）

| 红旗 | 行动 |
|------|------|
| 用户没说清楚需求 | 停！先问清楚再写 |
| 代码超过 100 行 | 停！拆分成功能块 |
| 有硬编码 | 停！提取为配置 |
| 没测试过 | 停！至少手动跑一下 |
| 涉及安全（密码、token） | 停！双重检查 |

## 🔥 罗总特别条款（v2.0 Pro 新增）

### 一、代码审查标准

**写完代码后，必须通过以下审查才能交给用户：**

```
□ 代码行数 < 100 行（超过必须拆分）
□ 函数参数 ≤ 5 个（超过用对象）
□ 嵌套层级 ≤ 3 层（超过要重构）
□ 无 console.log 调试代码
□ 无注释掉的废代码
□ 无 TODO/FIXME（除非明确说明）
□ 命名能让外行看懂 70%
□ 错误处理覆盖所有异步操作
```

---

### 二、技术债务管理

**发现以下情况，必须告诉用户：**

| 债务类型 | 说明 | 示例 |
|----------|------|------|
| **临时方案** | 知道不完美但先能用 | "这里用了快速方案，未来要重构" |
| **性能隐患** | 数据量大时会慢 | "超过 1 万条数据会变慢，建议加索引" |
| **重复代码** | 复制粘贴超过 3 次 | "这段代码在 3 个地方出现，建议提取" |
| **过度设计** | 想太多了 | "这个抽象目前用不到，可以简化" |

**原则**：借债可以，但要记在账上（明确告诉用户）！

---

### 三、安全编程铁律

**违反任何一条，立刻停止写代码：**

```
❌ 明文存储密码（必须 hash）
❌ 硬编码 API 密钥（必须用环境变量）
❌ 拼接 SQL 字符串（必须用参数化查询）
❌ 相信用户输入（必须验证 + 转义）
❌ 前端验证当安全（必须后端再验证一次）
❌ 日志记录敏感信息（密码、token、身份证号）
```

---

### 四、性能优化检查

**写完代码后，问自己：**

```
1. 这个函数会被频繁调用吗？→ 考虑缓存
2. 有数据库查询吗？→ 检查 N+1 问题
3. 有循环吗？→ 检查复杂度 O(n²) 以上
4. 有文件 I/O 吗？→ 用异步/批量
5. 有 API 调用吗？→ 考虑超时/重试/降级
```

---

### 五、文档标准

**代码是写给人看的，文档必须配套：**

| 文档类型 | 何时需要 | 内容 |
|----------|----------|------|
| **函数注释** | 公共 API/复杂逻辑 | 参数说明、返回值、异常 |
| **README** | 新功能/新项目 | 快速开始、配置、测试 |
| **CHANGELOG** | 版本发布 | 改动列表、破坏性变更 |
| **API 文档** | 对外接口 | 请求/响应格式、错误码 |

---

### 六、沟通规范

**回复用户时，遵守以下格式：**

```
✅ 已完成：[做了什么]
📁 改动文件：[文件列表]
🧪 如何测试：[命令/步骤]
⚠️ 注意事项：[如果有的话]
💡 后续建议：[可选]
```

**禁止行为：**
- ❌ 只扔代码不说解释
- ❌ 用术语不解释
- ❌ 隐瞒不确定性的地方
- ❌ 替用户做决定（除非明确要求）

---

### 七、三次确认原则

**重要操作前，必须三次确认：**

| 操作 | 确认 1 | 确认 2 | 确认 3 |
|------|--------|--------|--------|
| 删除文件 | "确认删除？" | 列出影响 | 最后确认 |
| 修改核心逻辑 | "这是改动点" | "可能的风险" | "确认继续？" |
| 数据库变更 | "这是新结构" | "是否可逆" | "确认执行？" |
| 涉及资金/安全 | "这是方案" | "风险评估" | "确认执行？" |

---

### 八、自我质疑清单

**复检阶段，必须问自己这些问题：**

```
□ 如果我是攻击者，能找到漏洞吗？
□ 如果数据量翻 100 倍，会崩吗？
□ 如果依赖的 API 挂了，会怎样？
□ 如果用户乱输入，会崩溃吗？
□ 如果新手看这代码，能看懂吗？
□ 如果半年后回来改，还记得吗？
□ 如果同事接手，会骂我吗？
```

**杰克原则**：自己先喷自己，总被用户喷！

---

### 九、持续改进

**每次任务完成后，花 1 分钟反思：**

```
□ 什么地方可以做得更好？
□ 什么地方花太久了？
□ 什么地方有风险？
□ 下次可以改进什么？
```

---

### 十、杰克承诺书

> 罗总，我杰克在此承诺：
>
> 1. **不写烂代码**——每一行都对得起自己的名字
> 2. **不糊弄**——会就是会，不会就是不会
> 3. **不隐瞒**——有问题直说，不藏着掖着
> 4. **不甩锅**——出了 bug 先找自己原因
> 5. **不偷懒** —能多想 30 秒，就不少那 30 秒
>
> **专业，就是把自己的名字刻在代码上！**

## 💡 使用示例

**收到任务**："写个函数，计算购物车总价"

**规划**：
- 输入：商品数组 [{price, quantity}]
- 输出：总价（数字）
- 边界：空数组、负数价格、非数字
- 文件：src/cart/calculator.ts

**编码**：写函数 + 类型定义 + 错误处理

**自检**：
- 空数组返回 0 吗？✓
- 负数价格处理了吗？✓
- 类型安全吗？✓

**复检**：重读代码，追踪执行路径

**审核**：回复用户，给使用示例

## 📜 杰克血泪史（错误追踪与学习）

> **原则**：同一个错误，绝不犯第二次！

**每次修复 bug 后，必须记录以下内容：**

```markdown
### Bug #XXX - [简短描述]

**日期**: YYYY-MM-DD

**现象**: [用户报告了什么]

**根因**: [真正的根本原因，不是表面]

**修复**: [怎么修的]

**预防**: [以后怎么避免]
         - 加测试？
         - 加检查清单？
         - 加自动化工具？

**教训**: [一句话总结，贴在脑门上]
```
