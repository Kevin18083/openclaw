# TDD 工作流技能

> 来源：ECC tdd-workflow skill + 杰克测试框架 v9.0

---

## 触发词

```
用 TDD 方式、测试驱动、先写测试、TDD 工作流
```

---

## 核心原则

### 1. 测试 BEFORE 代码
永远先写测试，再实现代码让测试通过。

### 2. 覆盖率要求
- 单元测试覆盖率 ≥ 80%
- 关键路径必须有集成测试
- 核心用户流程有 E2E 测试

### 3. 测试金字塔
```
        /‾‾‾‾‾‾\
       /  E2E   \      少而精（10%）
      /----------\
     /  集成测试  \    适中（20%）
    /--------------\
   /   单元测试     \  大量（70%）
  /------------------\
```

---

## TDD 五步流程

### 步骤 1: 写用户故事
```
作为 [角色], 我想要 [功能], 以便于 [价值]

示例：
作为用户，我想要搜索市场，
以便于找到相关的交易对手。
```

### 步骤 2: 生成测试用例
```javascript
describe('市场搜索', () => {
  it('返回相关结果', async () => { ... })
  it('空查询返回空数组', async () => { ... })
  it('超时后优雅降级', async () => { ... })
})
```

### 步骤 3: 运行测试（应该失败）
```bash
npm test
# 预期：测试失败 - 还没实现
```

### 步骤 4: 实现最小代码
```javascript
// 只写让测试通过的最少代码
async function search(query) {
  if (!query) return []
  // 实现...
}
```

### 步骤 5: 重构
- 删除重复代码
- 优化命名
- 提取公共逻辑
- 再次运行测试验证

```bash
# 运行测试
bash unified-gateway.sh --full .
```

---

## 检查清单

执行 TDD 时对照检查：

- [ ] 测试用例先于实现代码编写
- [ ] 测试覆盖了边界情况
- [ ] 测试覆盖了错误处理
- [ ] 测试可以独立运行
- [ ] 测试命名清晰（it('...'...)）
- [ ] 没有硬编码值
- [ ] 没有 TODO 注释
- [ ] 重构后测试依然通过

---

## 与杰克系统集成

1. **执行任务** → 扎克按照 TDD 五步流程执行
2. **提交检查** → `node jack-review.js submit "TDD 任务完成"`
3. **AI 审核** → 杰克 DeepSeek 检查测试覆盖率和代码质量
4. **输出结果** → ✅ 杰克已阅 / ❌ 需要补充测试
