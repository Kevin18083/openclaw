### 案例 001：Rush.js 增量构建 CI

**成功日期**：2026-03-09

**来源文件**：`ci.yml` (增量 CI 工作流)

**成功内容**：
```yaml
# 变更文件检测
- uses: tj-actions/changed-files@v45
  id: changed-files

# 生成缓存文件
- name: Process changed files
  run: |
    all_files="${{ steps.changed-files.outputs.all_changed_files }}"
    filtered_files=$(echo $all_files | sed 's/common\/changes/.*//')
    echo "[$(echo "$filtered_files" | sed 's/ /", "/g')]" > changed-files-cache.json

# Rush 增量构建
- name: Increment Build
  run: node common/scripts/install-run-rush.js increment --action build -p changed-files-cache.json
```

**成功关键**：
1. 只检测变更文件，不全网扫描
2. 过滤 `common/changes` 目录（Rush 变更文件）
3. 生成 JSON 缓存传递给后续 jobs
4. 使用 `rush increment` 只构建变更包

**使用场景**：
- Monorepo 项目 CI
- Rush.js 管理的项目
- 需要加速 CI 流程

**对比错误案例**：
- 错误案例：`ci@main.yml` 全量构建，耗时长
- 成功关键：增量构建，只编译变更的包

### 案例 004：License Header 自动添加脚本

**成功日期**：2026-03-09

**来源文件**：`add-license-header.sh`

**成功内容**：
```bash
# 检测是否已有许可证头
has_license_header() {
    local file="$1"
    local ext="${file##*.}"

    case "$ext" in
        "go"|"ts"|"tsx"|"js"|"jsx")
            head -n 5 "$file" | grep -q "Copyright.*coze-dev Authors"
            ;;
        "sh")
            head -n 10 "$file" | grep -q "Copyright.*coze-dev Authors"
            ;;
    esac
}

# 添加许可证头
add_license_header() {
    local file="$1"
    local temp_file=$(mktemp)

    case "$ext" in
        "go"|"ts"|"tsx"|"js"|"jsx")
            echo "$APACHE_LICENSE_HEADER" > "$temp_file"
            cat "$file" >> "$temp_file"
            ;;
        "sh")
            if head -n 1 "$file" | grep -q "^#!"; then
                head -n 1 "$file" > "$temp_file"  # 保留 shebang
                echo "$SHELL_LICENSE_HEADER" >> "$temp_file"
                tail -n +2 "$file" >> "$temp_file"
            fi
            ;;
    esac

    mv "$temp_file" "$file"
}
```

**成功关键**：
1. 先检查是否已有许可证头，避免重复添加
2. Shell 脚本保留 shebang (`#!/bin/bash`)
3. 使用临时文件，安全替换
4. Git Hook 自动重新暂存

**使用场景**：
- 开源项目许可证管理
- 企业代码规范要求
- Git 提交前检查

**复用建议**：
- ✅ 直接用：许可证检测逻辑

# 排除模式
EXCLUDE_PATTERNS=(
  '**/pnpm-lock.yaml'
  '**/__tests__/**'
  '**/*.snap'
  'common/changes/**'
)

# 转换为 Git 排除字符串
for pattern in "${EXCLUDE_PATTERNS[@]}"; do
  EXCLUDE_STRING+=":(exclude)$pattern "
done

# 获取变更文件
files=$(git diff --name-only --diff-filter=AM "origin/$TARGET_BRANCH..." $EXCLUDE_STRING)

# 检查文件体积
for file in $files; do
  file_size_kb=$((file_size / 1024))
  if [ "$file_size_kb" -gt "$size_limit" ]; then
    large_files_info+="- \`$file\` ($file_size_kb KB)\n"
  fi
done

### 案例 006：测试框架多轮重试机制

**成功日期**：2026-03-09

**来源文件**：`test-framework.sh` (杰克测试框架)

**成功内容**：
```bash
run_test() {
    local test_name="$1"
    local test_cmd="$2"
    local retry=1

    while [ $retry -le $MAX_RETRIES ]; do
        log_info ">>> 第 $retry 轮测试 (最大 $MAX_RETRIES 轮)"

        if eval "$test_cmd" >> "$MAIN_LOG" 2>&1; then
            log_success "$test_name - 通过 (第 $retry 轮)"
            PASSED_TESTS=$((PASSED_TESTS + 1))
            return 0
        else
            log_warning "第 $retry 轮失败，准备重试..."
            retry=$((retry + 1))

            if [ $retry -le $MAX_RETRIES ]; then
                log_info "等待 2 秒后重试..."
                sleep 2
            fi
        fi
    done

    # 所有轮次都失败，记录错误
    log_error "$test_name - 失败"
    record_error_to_memory "$test_name" "$test_cmd" "测试失败"
    FAILED_TESTS=$((FAILED_TESTS + 1))
    return 1
}
```

**成功关键**：
1. 默认 5 轮重试，可配置
2. 每轮之间等待 2 秒，避免瞬时故障
3. 详细日志记录每一轮结果
4. 失败自动记录到错误记忆库

**使用场景**：
- 测试不稳定功能（网络、数据库）
- CI/CD 自动化测试
- 需要重试机制的场景

**复用建议**：
- ✅ 直接用：多轮重试逻辑
- ✅ 直接用：日志记录机制
- ✅ 直接用：错误自动记录
