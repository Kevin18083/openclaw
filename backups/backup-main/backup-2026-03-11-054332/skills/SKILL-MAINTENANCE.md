# 技能维护指南

> **原则**：技能文档不是写完就结束的，需要持续更新才能保持价值！
>
> **目标**：让技能文档随着经验积累而"成长"！

---

## 📅 更新周期

### 每日（每次写代码后）

| 时间 | 任务 | 文件 |
|------|------|------|
| 写完代码后 | 过一遍安全检查清单 | code-security-check/SKILL.md |
| 遇到新错误 | 记录到错误日志 | tech-error-log/errors/*.md |
| 修复 Bug 后 | 记录教训 | code-quality-workflow/bug-lessons.md |

---

### 每周（建议周日）

| 任务 | 说明 | 预计时间 |
|------|------|----------|
| 回顾本周错误记录 | 有没有重复犯的错误？ | 10 分钟 |
| 整理错误日志 | 分类归档，添加关键词 | 15 分钟 |
| 检查安全规范 | 有没有新的漏洞需要关注？ | 15 分钟 |

**周检查清单**：
```
□ 本周新增了多少条错误记录？
□ 哪类错误最多？（找到自己的薄弱点）
□ 有没有重复犯同样的错误？
□ code-security-check 需要补充新的检测模式吗？
□ 哪个技能帮助最大？
```

---

### 每月（建议月末）

| 任务 | 说明 | 预计时间 |
|------|------|----------|
| 安全规范更新 | 检查 OWASP Top 10、CVE 漏洞库 | 30 分钟 |
| 恶意代码模式更新 | 补充新的攻击手法 | 30 分钟 |
| 技能文档审查 | 有没有过时的内容需要更新？ | 30 分钟 |
| 统计与回顾 | 本月进步了多少？ | 15 分钟 |

**月检查清单**：
```
□ 本月新增了多少条错误记录？
□ 哪个技术栈的坑最多？
□ 有没有系统性改进？（比如加测试、改流程）
□ 需要补充哪些知识盲区？
□ 安全规范有更新吗？（OWASP、CVE）
□ 有新的恶意代码模式吗？
□ 技能文档需要重构吗？
```

---

### 每季度（每 3 个月）

|------|------|----------|

## 🔄 更新触发条件

### 必须更新的情况

| 场景 | 行动 |
|------|------|
| 遇到新的错误类型 | 立刻记录到 tech-error-log |
| 发现新的安全漏洞 | 更新 code-security-check 的 Module C |
| 发现新的恶意代码模式 | 更新 code-security-check 的 Module B |
| 修复了一个复杂的 Bug | 记录到 bug-lessons.md |
| 学到了新的安全最佳实践 | 更新 Module C |
| OWASP 发布新指南 | 审查并更新相关章节 |
| 使用的框架有重大安全更新 | 审查并更新相关章节 |

---

### 可以更新的情况

| 场景 | 行动 |
|------|------|
| 某条错误记录帮助了多次 | 添加到"常见错误速查" |
| 某个检查项从未用到 | 考虑删除或合并 |
| 文档结构变得混乱 | 重组分类 |
| 有新工具可以自动化检查 | 添加到工具推荐 |

## 🛠️ 自动化检查工具

### 依赖安全扫描

```bash
# npm 项目
npm audit
npx audit-ci --moderate  # 中等以上漏洞才失败

# 生成报告
npm audit --json > audit-report.json
```

### 代码安全扫描

```bash
# ESLint 安全插件
npm install eslint-plugin-security
eslint --plugin security .

# SonarQube (大型项目)
# https://www.sonarqube.org
```

### 定期运行

```json
// package.json
{
  "scripts": {
    "security:audit": "npm audit && eslint --plugin security ."
  }
}
```

| 指标 | 健康状态 | 警戒状态 |

## 💡 维护技巧

### 1. 趁热打铁

```
遇到错误 → 解决后**立刻**记录
          ↓
      记忆新鲜，记录详细
          ↓
      下次查阅时更有价值
```

### 2. 宁多勿缺

```
不确定要不要记录？→ 记下来！
                   ↓
               以后可以删除
                   ↓
               但删除了就找不回了
```

### 3. 关键词标记

```
每条记录添加 3-5 个关键词
             ↓
         方便以后搜索
             ↓
         grep -r "关键词" errors/
```

### 4. 定期清理

```
每季度审查一次：
- 删除重复的记录
- 合并相似的内容
- 更新过时的信息
```

### 5. 版本控制

```
每次重大更新后 commit：
git add -A
git commit -m "docs: 更新 XXX 技能 - 添加 YYY 内容"
```

## 🚨 紧急情况处理

### 发现严重安全漏洞

```
1. 立即修复代码
   ↓
2. 记录到 bug-lessons.md（具体项目）
   ↓
3. 更新 code-security-check 的 Module C（通用规范）
   ↓
4. 通知罗总/扎克
   ↓
5. 检查其他项目是否有同样问题
```

### 发现新的恶意代码模式

```
1. 分析攻击手法
   ↓
2. 记录到 code-security-check 的 Module B
   ↓
3. 更新检查清单
   ↓
4. 审查现有代码是否有风险
```
