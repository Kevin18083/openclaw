> **版本**: v2.0
> **创建时间**: 2026-03-09
> **更新**: 2026-03-09 添加修改循环机制
> **目标**: 扎克执行任务，杰克检查，有问题扎克自己改，通过后汇报罗总

## 🎯 触发条件

**罗总说以下任意关键词**，触发检查流程：
- ✅ "让杰克检查"
- ✅ "给杰克检查"

**没有说这两个词** → 扎克自己完成，不触发检查。

---

## 📋 核心原则

| 角色 | 职责 | 原则 |
|------|------|------|
| **罗总** | 决策者 | 说关键词触发检查，接收最终结果 |
| **杰克** | 检查者 | 检查问题、教怎么改、**不动手改** |
| **扎克** | 执行者 | 执行任务、**自己修改**、重新提交、最终汇报 |

**关键点**：
1. 杰克只检查、只教，**不替扎克改代码**
2. 扎克有问题自己改，改完再提交
3. 循环直到通过，**中途不打扰罗总**
4. 只有最终通过了，扎克才向罗总汇报

---

## 🔄 阶段详情

### 阶段 1: 扎克执行任务

扎克接收任务并执行，完成后准备提交检查。

---

### 阶段 2: 触发检查

罗总说："让杰克检查" 或 "给杰克检查"

→ 触发检查流程

---

### 阶段 3: 杰克初检

**检查清单**：
```
□ 格式检查 - 报告格式是否完整
□ 步骤检查 - 步骤是否清晰、可追溯
□ 文件检查 - 修改的文件是否存在
□ 语法检查 - 代码/配置语法是否正确
□ 完整性检查 - 是否有遗漏
```

**输出格式**：
```markdown
## 初检结果

### ✅ 通过项
[列表]

### ❌ 问题项
1. [问题 1] - [位置] - [严重程度]
2. [问题 2] - [位置] - [严重程度]

### 📝 修改建议
[具体怎么改，附示例]

### 结论
□ 进入深检  /  □ 需要修改
```

---

### 阶段 4: 扎克修改（如有问题）

**杰克输出问题后，扎克负责**：
1. 按杰克的建议修改
2. 填写修改报告
3. 重新提交 "让杰克检查"

**修改报告格式**：
```markdown
## 修改报告

### 原问题
[引用杰克指出的问题]

### 修改内容
1. [修改 1]
2. [修改 2]

### 修改后内容
[粘贴修改后的代码/配置]

### 自验证
- [x] 已验证修改正确
- [x] 已测试相关功能

请杰克重新检查。
```

---

### 阶段 5: 杰克深检

**初检通过后，进行深度检查**：

```
□ 逻辑验证 - 业务逻辑是否正确
□ 边界检查 - 是否考虑边界情况
□ 错误处理 - 异常处理是否完善
□ 性能影响 - 是否有性能问题
□ 安全隐患 - 是否有安全风险
□ 兼容性 - 是否与现有系统兼容
□ 可维护性 - 代码是否易维护
□ 文档完整 - 文档/注释是否完整
```

**输出格式**：
```markdown
## 深检结果

### 🔍 深度检查项

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 逻辑验证 | ✅/⚠️/❌ | [说明] |
| 边界检查 | ✅/⚠️/❌ | [说明] |
| ... | ... | ... |

### 最终结论
□ ✅ 通过 - 可向罗总汇报
□ ⚠️ 有条件通过 - 建议优化
□ ❌ 不通过 - 需要修改

### 修改建议（如有）
[具体怎么改]
```

---

### 阶段 6: 循环修改（如有问题）

```
杰克指出问题 → 扎克修改 → 重新提交 → 杰克再检查
```

循环直到**杰克确认通过**。

---

### 阶段 7: 扎克向罗总汇报

**杰克检查通过后，扎克向罗总提交最终报告**：

```markdown
## 📋 任务完成报告

**任务名称**: [任务描述]
**执行时间**: [开始] - [完成]
**检查状态**: ✅ 杰克两轮检查通过

### 执行内容
[简要说明做了什么]

### 修改的文件
- [文件 1 路径]
- [文件 2 路径]

### 检查情况
- 初检：✅ 通过
- 深检：✅ 通过
- 检查人：杰克
- 迭代次数：[修改了几次]

### 自验证
- [x] 功能已测试
- [x] 无遗留问题

---
请罗总审阅。
```

## 📝 示例流程

### 场景：扎克修改缓存配置

**1. 扎克执行**
```
扎克：缓存配置改好了
```

**2. 罗总触发检查**
```
罗总：让杰克检查
```

**3. 杰克初检**
```markdown
## 初检结果

### ❌ 问题项
1. JSON 语法错误 - cache-config.json 第 5 行 - 高
   原因：缺少逗号

### 📝 修改建议
在第 5 行 "enabled": true 后面添加逗号：

修改前：
"enabled": true
"promptCache": {

修改后：
"enabled": true,
"promptCache": {

请扎克修改后重新提交检查。
```

**4. 扎克修改**
```markdown
## 修改报告

### 原问题
JSON 语法错误 - 缺少逗号

### 修改内容
已在第 5 行添加逗号

### 修改后代码
{
  "cacheOptimization": {
    "enabled": true,
    "promptCache": {
      "enabled": true,
      "ttl": 7200
    }
  }
}

### 自验证
- [x] 已验证 JSON 格式正确

请杰克重新检查。
```

**5. 杰克深检**
```markdown
## 深检结果

### 🔍 深度检查项

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 逻辑验证 | ✅ | 配置逻辑正确 |
| 语法检查 | ✅ | JSON 格式正确 |
| 完整性 | ✅ | 配置项完整 |

### 最终结论
✅ 通过 - 可向罗总汇报
```

**6. 扎克汇报罗总**
```markdown
## 📋 任务完成报告

**任务**: 修改缓存配置
**状态**: ✅ 杰克检查通过
**修改的文件**: cache-config.json
**迭代次数**: 1 次

请罗总审阅。
```

##  使用说明

### 罗总
1. 布置任务给扎克
2. 扎克完成后，说 "让杰克检查" 或 "给杰克检查"
3. 等待扎克最终汇报
4. 审阅通过或指出问题

### 扎克
1. 接收任务并执行
2. 罗总触发检查后，提交检查报告
3. 如杰克指出问题，按建议修改
4. 修改后重新提交 "让杰克检查"
5. 检查通过后，向罗总汇报最终结果

### 杰克
1. 收到检查请求后，进行初检
2. 初检有问题 → 输出问题和修改建议 → 等待扎克修改
3. 初检通过 → 进行深检
4. 深检有问题 → 输出问题和修改建议 → 等待扎克修改
5. 深检通过 → 通知扎克可以向罗总汇报
6. **不替扎克改代码，只教怎么改**
